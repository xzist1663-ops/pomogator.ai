# Pomogator backend (локальный, этап отладки)

Серверная часть Pro-функций расширения. Лежит в папке `backend/` **внутри того же
репозитория**, что и расширение. Запускается локально; на сервер переедет позже
сменой одного адреса.

Стек: Node + TypeScript (Express), Postgres (в Docker), Drizzle ORM.

## Почему так

- **Postgres** вместо SQLite — стандарт для многопользовательского сервиса:
  параллельные записи, доступ по сети, нормальные миграции. Поднимается в Docker,
  ставить на ПК ничего не нужно.
- **docker-compose** — поднимает Postgres (а потом и сам бэкенд) одной командой,
  и локальное окружение совпадает с будущим сервером.
- **Drizzle** — типобезопасные запросы и миграции на TypeScript.

## Запуск (по шагам)

Нужен установленный Docker Desktop (бесплатный).

```bash
cd backend
cp .env.example .env          # Windows: copy .env.example .env
# впиши в .env свои OZON_CLIENT_ID и OZON_API_KEY

docker compose up -d db       # поднять Postgres в контейнере
npm install
npm run db:push               # создать таблицы из schema.ts (один раз)
npm run dev                   # запустить бэкенд на localhost:3000
```

Ключи Ozon: кабинет → Настройки → API-ключи → **Seller API**, тип «Администратор».

Проверка:
```bash
curl http://localhost:3000/health
curl http://localhost:3000/api/products -H "X-Access-Token: dev-token"
```

Остановить Postgres: `docker compose down` (данные сохранятся в томе `pgdata`).

## Два режима запуска

- **Разработка (рекомендуется):** Postgres в Docker + бэкенд на хосте через
  `npm run dev` — быстрые правки, горячая перезагрузка. Это то, что выше.
- **Всё в контейнерах (ближе к проду):** раскомментируй сервис `backend`
  в `docker-compose.yml`, поставь в `.env` `DATABASE_URL` с хостом `db` вместо
  `localhost`, затем `docker compose up`.

## Эндпоинты

Все, кроме `/health`, требуют заголовок `X-Access-Token: dev-token`.

- `GET  /api/products` — экономика по всем товарам (цена, комиссия, логистика,
  себестоимость, маржа, цвет `green|yellow|red|no_cost`). Расширение матчит строки
  таблицы по `offerId`.
- `POST /api/cost` — записать себестоимость: `{ "offerId": "YNG200", "cost": 120 }`.
- `GET  /api/profit` — финансовый нетто по периодам (today/yesterday/week/month)
  + дельта сегодня vs вчера.

## Что важно знать (честные оговорки)

1. **Формы ответов Ozon надо сверить на твоём кабинете.** `/v5/product/info/prices`
   и `/v3/finance/transaction/list` версионируются; имена полей в `commissions{}`
   и обёртка ответа могут отличаться. Клиент в `src/ozon.ts` написан защитно, но
   первый прогон почти наверняка потребует подправить пути полей.
2. **`/api/profit` — это v1:** сумма финансовых операций за период, без вычета
   себестоимости проданного. Честнее оборота, но не итоговая прибыль. Вычет COGS
   добавим, когда увидим реальную форму транзакций.
3. **Таймзона.** Сутки считаются в UTC; Ozon живёт по Москве — на стыке суток
   цифры могут «съезжать». Поправим на МСК после сверки.
4. **Скидка Ozon за свой счёт** здесь НЕ считается — её берём из DOM кабинета
   в расширении (цена покупателя через API больше не отдаётся).

## Подключение из расширения

Контент-скрипт на `seller.ozon.ru` не ходит на `localhost` напрямую
(mixed-content): запросы идут через **background service worker**, в манифесте —
`host_permissions: ["http://localhost:3000/*"]`. Адрес бэкенда держи в одной
константе `API_BASE`, чтобы при переезде на сервер менять одну строку.
